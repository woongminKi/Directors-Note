# 2026-06-05 소비자 결제 (카카오페이, pay-to-unlock) — D-③ 선행 토대

D-③(평가자 정산/환불)의 **선행 토대 = 소비자 실결제**를 구현. 카카오페이 단건결제(리다이렉트)로
주문/거래 모델 + provider 추상화를 만들고, 기존 `pay-to-unlock`(채점 후 결제→결과 공개) 게이트를
유지했다. 정산·환불은 후속(인터페이스 `cancel()` 자리만). sandbox(`TC0ONETIME`)로 구현·검증.

## 배경

소비자 결제는 stub(`submissions.paid_at` 1개, 금액·거래 없음)뿐이었다. 정산·환불은 "돈이 실제로
들어오고 금액/거래가 기록되는" 토대가 있어야 얹힌다. 결제 provider 는 **카카오페이로 확정**
([[payment-provider-kakaopay]] — 영세 카드율 0.20% 우위).

## 만든 것

수직: **scored 소비자 "결제하기" → payReady(주문 생성+ready) → 카카오페이 결제창 → 승인 콜백 → approve → paid_at 스탬프 → release(결과 공개)**

| 영역 | 파일 | 내용 |
|---|---|---|
| env | `src/lib/env.ts` | `KAKAO_PAY_SECRET_KEY`, `KAKAO_PAY_CID`(optional) |
| DB | `migrations/0020_payment_orders.sql`, `0021_*_rls.sql`, `schema.ts` | `payment_orders`(주문/거래) + RLS(본인 SELECT) |
| config/타입 | `src/lib/payments/{config,types}.ts` | `SUBMISSION_PRICE_KRW=9900`, `PaymentProvider` 인터페이스 |
| provider | `kakaopay-provider.ts`, `stub-provider.ts`, `factory.ts` | KakaoPay ready/approve, Stub(무료 파일럿), factory |
| actions | `src/lib/payments/actions.ts` | `payReady`, `approveOrder`(멱등, paid_at 스탬프+release) |
| 콜백 | `src/app/api/payments/kakao/approve/route.ts` | 승인 콜백(approval_url) |
| UI | `src/app/(consumer)/submissions/[id]/pay-button.tsx` | payReady redirect 흐름 |

### 핵심 설계
- **금액 서버결정**(`SUBMISSION_PRICE_KRW`), 주문 레코드에 저장(과거 주문 불변). 클라 금액 신뢰 X.
- **멱등**: `approveOrder` 는 이미 approved 면 no-op. `paid_at` 은 null 일 때만 스탬프.
- **순서**: provider.approve 성공 → order approved → paid_at 스탬프 → release. 실패 시 order failed, release 안 함.
- **stub 모드**(`FEATURE_PAYMENT_ENABLED=false`): payReady 가 외부 결제창 없이 즉시 approve→stamp→release
  → 기존 무료 파일럿 동작과 동치. 기존 `payForSubmission`(payment-action.ts)은 레거시로 보존.
- `submissions.paid_at` 은 unlock 신호로 유지, 금액/거래 상세는 `payment_orders`.
