# 2026-06-04 영상분석 로직 진단 및 LLM-judge 개선 플랜

## 배경
어제(06-03) 다른 컴퓨터에서 reference 영상 6개를 3파트 구조로 재임베딩(seed)함.
영상은 학생 실기 평가본으로, 연기·몸짓·발성을 보려고 **멀리서 촬영** → 얼굴이 작게 잡힘.
"어떤 기준·로직으로 분석하는지" 파악 + 현재 로직 검증 + 개선 플랜 수립이 목표.

- 임베딩 시각 06-03 07:52~07:57 UTC. 6개 모두 `scene_type=modern_monologue`, tier A×2/B×2/C×2.

## 분석 로직 (현재 v1, 코드 기준)
파일: `src/lib/evaluation/{vertex.ts,grade-derivation.ts,gpt-4o-mini-letter.ts}`, `migrations/0013_*.sql`
1. 학생 영상 3파트 분할 임베딩(Vertex `multimodalembedding@001`, 1408d).
2. 파트별 cosine 매칭(RPC `search_reference_matches_by_part`, 같은 part끼리 top-5).
3. 점수 = `TIER_BASE[매칭ref등급] + (cos-0.5)*3` clamp 0~10. TIER_BASE={A:8,B:6.5,C:5,D:3.5}.
4. part1→표정(expression), part2→발성(vocal), part3→입시완성도(examReadiness).
5. 등급 = 3축 평균: A≥7.25 / B≥5.75 / C≥4.25 / else D.
6. gpt-4o-mini는 **채점 안 함** — 점수·등급을 받아 부모 letter만 작성. `llm_as_judge` 경로는 함수만 있고 미연결(TODO D12).

→ **핵심: "연기를 평가"하는 게 아니라 "사전 등급이 매겨진 reference와 닮음(cosine)"만 봄.** 축 이름은 시간대 라벨일 뿐.

## 실측 검증 결과 (결정적)
- WITHIN-PART cosine(다른 영상 같은 part) ~0.88, CROSS-PART(같은 영상 다른 part) ~0.91.
- 같은 tier 쌍 ≈ 다른 tier 쌍 → **cosine에 등급 변별력 없음**.
- 6개를 각각 "학생"으로 넣고 production 공식 그대로 적용 → **tier match 1/6 = D13 환산 1.7/10** (게이트 ≥7/10 압도적 미달).
- 등급이 전부 A로 붕괴. 실제 C급이 A급 ref에 cos 0.91~0.95로 매칭 → A.
- 원인: 원거리 촬영으로 표정·발성 미세신호가 1408d 전체프레임 임베딩에서 장면·전신·배경에 묻힘 + cosine은 "품질"이 아닌 "장면 유사도"를 잼.
- 캐비엇: n=6, reference끼리 비교라 통계 단정 불가. 단 "자기 reference의 A/B/C조차 못 가른다"는 구조적 한계 입증.

### 재현성(정합성) 요구사항 — 사용자 추가 요청 (2026-06-04)
**요구: 같은 영상은 언제·어느 머신에서 돌려도 같은 임베딩/점수가 나와야 한다.** (provenance ③는 사용자 확인 완료 — 동일 원본 업로드)
관찰된 "같은 영상 점수 차이"의 유력 원인: v0(0-120s 단일창) → v2(3파트) **방법론 변경** 또는 Vertex 비결정성. 둘 다 대응 필요.
- **임베딩 계층**: 모델 버전 핀(`multimodalembedding@001` 고정), 리전 고정, part window 고정. **Vertex video 임베딩 결정성 1회 실측**(같은 파일 2회 임베딩 → cos≈1.0 확인). 임베딩은 이미 `cacheEvaluationEmbedding`으로 캐시됨.
- **judge 계층(신규 primary, 재현성 핵심)**: LLM은 기본 stochastic → `temperature=0`/결정적 디코딩, 프롬프트·루브릭 **버전 고정**, structured-output 스키마 강제, **(영상해시, 모델버전, 루브릭버전)별 점수 캐시**(재호출 시 동일 결과), borderline만 self-consistency.
- **provenance 스탬프**: `rawResponseJson`에 모델버전·루브릭버전·임베딩방법 기록 → 재실행 비교 가능, 점수 drift 추적.
- **Phase 0 하니스에 재현성 지표 추가**: 같은 입력 N회 → 점수 분산 ≈ 0 검증(게이트). tier-match와 별개 축.

**결정성/재현성 실측 결과 (2026-06-04, downloads/상1.mp4 재임베딩):**
- **Vertex 결정성: 완전 결정적.** 같은 파일 2회 호출(A vs B) → 3파트 모두 cos = **1.000000**.
- **재현성: 완벽.** 지금 임베딩 vs DB(어제 다른 컴퓨터) → 상1이 DB #1A와 3파트 모두 cos = **1.0000** 정확 일치. (부수 확인: #1A=상1, 시드가 파일명 순서대로 들어감)
- **결론: 임베딩 계층 재현성 문제 없음(해결).** 관찰됐던 "같은 영상 점수 차이"는 비결정성이 아니라 **방법론 변경(v0 단일창 → v2 3파트)** 때문. 모델버전·리전·part window 핀 고정 + 기존 임베딩 캐시면 충분, 추가 완화 불필요.
- **→ 재현성 리스크는 이제 전적으로 judge 계층(LLM stochastic)에 집중.** temperature=0 + 프롬프트/루브릭 버전핀 + 점수 캐시 + borderline self-consistency 가 재현성 작업의 실체. Phase0 재현성 게이트는 judge 점수 분산 검증에 집중.
- 실측 스크립트: `/tmp/determinism.ts` (세션용, Vertex 6콜 ≈ $0.006 소요).

## Phase 1 조기 검증 결과 (2026-06-04, 미커밋)
Gemini judge 구현 + 6영상(상/중/하 라벨) 조기 검증 완료. 신규: `src/lib/evaluation/{gcs-staging.ts, llm-judge.ts, prompts/judge-rubric-v1.ts}`, `scripts/calibration-judge.ts`(+`calibrate:judge`). 하니스 seam에 `makeJudgeEvaluator` 배선. cosine 무손상(여전히 1/6).
- **결과: 2/6 = 3.3/10 ❌ FAIL** (gemini-2.5-flash, full video, temp=0). cosine 1/6보다 미세 우위지만 **착시** — 전부 A로 붕괴(B·C도 A 판정). 맞은 2개는 A 2개뿐.
- **원인: 절대평가 leniency/천장효과.** 6개 다 "숙련된 연기"로 읽고 전 축 7.0~9.5 앵커. rationale는 on-rubric(발성=오디오 인용, 신체로 정서 추론, 무용 통제 묘사)로 양호하나, 절대 밴드라 중/하를 끌어내릴 상대 기준이 없음. C#2에 노래 환각(긍정 서술 경향).
- 축별: 몸짓 최고 인플레(8.0~9.5), 표정만 유일하게 spread(7.0~9.0)로 변별에 근접.
- gemini-2.5-**pro는 full video 불가**(입력 65,536 토큰 cap < 영상 ~77k → 400). **flash(1M)가 기술적 정답.**
- latency ~28s/영상, 비용 6콜 <$0.5.

**전략 전환:** 문제는 "judge가 연기를 못 읽음"이 아니라 **"변별(discrimination) 방법"**. judge 배관은 production-shaped로 견고. 개선 레버: ①few-shot 앵커(알려진 A/B/C 예시 주입) ②forced-ranking/pairwise 비교(독립 절대점수 대신) ③코치 분포에 맞춘 밴드 임계 재보정. ①③은 코치 ground-truth가 전제.
