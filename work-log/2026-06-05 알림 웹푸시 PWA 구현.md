# 2026-06-05 알림 — 공통토대 + 웹푸시 + PWA 구현 (Phase A 잔여 D-②)

Phase A 잔여 작업 D 중 **알림(D-②)** 의 첫 사이클을 구현. 채널 불문(channel-agnostic) 알림
공통토대 + **웹 푸시** 채널 + **PWA 설치형**까지. 카카오 **알림톡은 인터페이스 stub**(미발송),
네이티브 앱(RN/Capacitor)은 후속 사이클. 토대는 두 후속을 어댑터로 받도록 설계 — 재작업 없음.

## 배경 / 결정

- "앱 푸시"는 네이티브 앱이 없으므로 **웹 푸시**(Service Worker + VAPID)로 구현. iOS 는 홈화면
  설치(PWA) 시 16.4+ 부터 푸시 지원, Android/데스크톱은 바로 동작.
- **알림톡은 지금 발송 불가**: 전화번호 미수집 + 카카오 비즈채널·사업자·대행사·템플릿 심사 선행
  필요(D-③ 결제 provider 처럼 외부 의존). 그래서 어댑터 인터페이스만 두고 stub.
- 전달 모델: **아웃박스 + `after()` 즉시시도 + daily cron 재시도**(Hobby 는 daily cron 만 허용).

## 만든 것

수직: **이벤트(release/assign/score) → enqueue(아웃박스) → after() 즉시 dispatch → 채널 어댑터 발송**, 실패/누락은 daily cron 재시도.

| 영역 | 파일 | 내용 |
|---|---|---|
| DB | `migrations/0018_notifications.sql`, `0019_notifications_rls.sql`, `schema.ts` | `push_subscriptions`(구독), `notifications`(아웃박스) + RLS |
| env | `src/lib/env.ts` | `FEATURE_WEB_PUSH`, `VAPID_*`, `NEXT_PUBLIC_VAPID_PUBLIC_KEY`, 알림톡 stub key |
| 토대 | `src/lib/notifications/{types,copy,factory,actions}.ts` | 타입·P2 문구·factory·enqueue/dispatch/drain/notify/savePushSubscription |
| 채널 | `web-push-channel.ts`, `alimtalk-channel.ts` | WebPush 실구현(죽은 구독 정리), AlimTalk stub |
| cron | `src/app/api/cron/dispatch-notifications/route.ts`, `vercel.json` | 재시도 드레인(daily `0 19 * * *` = KST 04:00) |
| 훅 | `release-action.ts`, `score-action.ts`, `assignment/actions.ts` | 결과공개/채점완료(소비자), 새 배정(평가자) |
| 클라 | `public/sw.js`, `components/notifications/push-opt-in.tsx`, `components/pwa/install-prompt.tsx` | SW, 권한·구독 UI, 설치 프롬프트 |

### 이벤트별 문구 (점수/등급 미포함 — P2 하드게이트)
- `submission_released` → 소비자 "결과가 준비됐어요" → `/submissions/[id]`
- `submission_scored` → 소비자 "채점이 끝났어요 / 결제 후 결과 공개" → `/submissions/[id]`
- `evaluator_assigned` → 평가자 "새 채점 배정 / 48시간 내" → `/score/[id]`

### 핵심 설계
- **실패 격리**: `notify()` 가 enqueue+dispatch 를 try/catch 로 감싸 호출 액션(release/score/assign)을
  절대 깨지 않음. `after()` 콜백 오류도 `.catch` 로 무시.
- **멱등**: `dispatchNotification` 은 이미 `sent` 면 재발송 skip. 즉시시도/cron 재시도 중복 안전.
- **죽은 구독 정리**: WebPush 404/410 → 해당 `push_subscriptions` 삭제.
- **flag off**: `FEATURE_WEB_PUSH≠true` 면 web_push enqueue 자체 skip(영구 미발송 행 방지).

### ⚠️ 테스트 실행 주의 (중요)
DB-gated 통합테스트는 **단일 공유 dev DB** 를 쓰므로 **반드시 `--no-file-parallelism` 로 직렬 실행**할 것.
병렬로 돌리면 assignment 계열 테스트들이 같은 평가자 풀에 동시 시드해 `cron-sweep` 의 "특정 평가자로
재배정" 단언이 깨진다(코드 버그 아님, 테스트 격리 문제). 권장 커맨드:
```
set -a; . ./.env.local; set +a
RLS_TEST_DB=1 ASSIGNMENT_TEST_DB=1 bun run test:ci tests/integration --no-file-parallelism
```
> 구현 중 이전 중단 실행의 `rlstest+` 잔여 평가자가 풀을 오염시켜 테스트가 실패 → 사용자 승인 후
> `rlstest+%` 테스트 데이터(public+auth.users 5명, 알림 8건; 실제 데이터 0)를 정리해 해소.

  (`sql.unsafe(file).simple()`, `max:1`).
- [ ] 배포 후 **Cron Jobs** 에 `/api/cron/dispatch-notifications` daily 등록 확인.
- [ ] 동작 확인: 소비자/평가자 로그인 → "알림 켜기" → 권한 허용 → 테스트 이벤트(평가자 배정 등) →
  푸시 수신. **iOS 는 홈화면 설치(PWA) 후** 확인.

## 남은 D 작업 / 후속

- **알림톡 실발송**: `AlimTalkChannel` 구현(대행사 HTTP) + 전화번호 수집(가입/인테이크) +
  카카오 비즈채널·템플릿 심사. 채널 enum·아웃박스·dispatch 그대로 재사용.
- **네이티브 앱**: RN/Capacitor 래퍼 + `FcmChannel`/`ApnsChannel` 추가(동일 추상화에 채널만 추가).
- **D-③ 평가자 정산/환불**: C 의 결제 provider 결정(Toss/카카오페이) 대기.
